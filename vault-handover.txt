FROZEN — Architecture & Decisions
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

CORE ARCHITECTURE
-----------------
- 6-digit PIN → PBKDF2 key → AES-256-GCM encrypts entry array → localStorage
- localStorage keys: vault_data, vault_pin, vault_theme, vault_bio,
  vault_bio_cred, vault_lockout, vault_fails, vault_last_backup
- state{} holds: entries, currentPIN, unlocked, unlockedViaBio,
  stepUpDone, pinBuffer, pinMode, pinTemp, editId, activeFilter,
  currentNav, detailSource, failCount, preLockScreen, formDraft,
  bioGen, bioPending
- Auto-lock: 2min idle. Lockout: 5 fails → timed delay.
- Single file always. No build tooling. No frameworks.

ENTRY SCHEMA
------------
All:      id, type, name, fav, created, updated, customFields[], sensitiveOverride
app:      url, username, password, notes
wifi:     ssid, password, securityType, notes
note:     note
card:     cardHolder, cardNumber(raw digits only), expiry(MM/YY), cvv, pin, notes
bank:     bankName, accountNo, ifsc, accountType, netBankingId, netBankingPw, notes
identity: docType(aadhaar/pan/passport/dl/voterid/other), docName,
          docNumber(raw digits for aadhaar), idName, dob(DD/MM/YYYY),
          expiry(DD/MM/YYYY), nationality, issuingState, notes

BIOMETRIC RULES
---------------
- WebAuthn platform only. Credential ID stored in vault_bio_cred.
- Fresh app open: PIN required once to seed currentPIN in memory.
  Bio button shows but does NOT auto-trigger on fresh open.
  Subtitle shows "Enter PIN once to activate biometric".
- After first PIN entry: bio auto-triggers on every subsequent auto-lock.
- Auto-trigger mechanism: initPINScreen('unlock') sets state.bioPending=true
  (does NOT fire immediately via timer — WebAuthn requires document focus,
  and visibilitychange/lockVault fire BEFORE focus returns on Android Chrome).
  triggerPendingBio() is called from visibilitychange(visible) and
  window 'focus' listeners; only proceeds if document.hidden===false AND
  document.hasFocus()===true at that moment, then fires after 150ms.
  state.bioGen exists but bioPending model makes races largely moot.
- unlockedViaBio=true → step-up PIN required for Card/Bank/Identity view,
  for Add/Edit/Delete/Export actions, AND for restoring an in-progress
  Add/Edit form after re-auth (restoreAfterUnlock form-screen branch).
- stepUpDone=true after first step-up (resets on lockVault).
  Manual PIN unlock (not bio) ALSO sets stepUpDone=true — re-entering
  PIN to unlock proves PIN knowledge, satisfies step-up for that session.
- Settings security actions always prompt PIN directly (bypass stepUpDone).
- Step-up condition: state.unlockedViaBio (not isBioEnabled).

SCREEN RESTORE AFTER RE-AUTH
----------------------------
- lockVault() captures state.preLockScreen (form-screen/detail-screen/
  fav-screen/settings-screen) before locking. For form-screen, also
  captures state.formDraft via captureFormDraft() — snapshots formState,
  customFieldsData, editId, and all #form-body .field-input values.
- restoreAfterUnlock() (async) is called after successful unlock
  (manual PIN or bio) instead of always going to home-screen:
  - form-screen: if unlockedViaBio, requires step-up PIN first
    (cancel -> discard draft, go home). Otherwise restores draft via
    renderForm() + restoreFormDraft().
  - detail-screen: re-opens same entry via openDetail() (which itself
    re-checks step-up for sensitive entries if unlockedViaBio).
  - fav-screen / settings-screen: just re-shown.
  - default: home-screen (cold starts always land here, since
    preLockScreen is only set by lockVault during an active session).
- Cold start (fresh page load) is unaffected — currentPIN empty,
  preLockScreen never set, always goes to home after unlock.

SECURITY TIERS
--------------
- Sensitive: Card, Bank, Identity (require step-up when bio unlock).
- Non-sensitive: App, WiFi, Note.
- Per-entry override via sensitiveOverride field.

NUMBER FORMATTING
-----------------
- cardNumber and Aadhaar docNumber stored as raw digits (no spaces).
- Formatted on display only: formatCardNumber(), formatAadhaar().
  15 digits → Amex 4-6-5. 16 digits → standard 4-4-4-4.
  12 digits (Aadhaar) → 4-4-4. Other lengths → shown as-is.
- Card list subtitle (•••• 1234) reads raw digits directly.

RULED OUT — PERMANENTLY
-----------------------
- Password generator: not part of Vault scope, ever.
- Multi-page architecture: single file is intentional for security.
- Storing PIN or decryption key in Firestore: PIN never leaves device.

R5 IMPLEMENTATION NOTES (shipped v4.3)
---------------------------------------
- Firebase loaded via CDN ES module dynamic import() (firebase-app,
  firebase-auth, firebase-firestore v10.12.2) — no build step, lazy
  loaded only when sync is touched.
- Auth: signInWithPopup (GoogleAuthProvider), falls back to
  signInWithRedirect on auth/popup-blocked or cancelled-popup-request.
- Firestore doc: users/{uid}/vault/data — fields {data: <encrypted
  blob string>, updated: <ms timestamp>}.
- New localStorage keys: vault_local_updated (set on every
  saveEntries()), vault_cloud_sync ('1'/'0'), vault_cloud_uid,
  vault_cloud_email, vault_last_sync, vault_sync_pending (transient —
  set before signInWithRedirect, cleared on return; signals
  restore() to call getRedirectResult()).
- Settings → Cloud Sync section:
  - Toggle ON: signs in (popup), then reconciles local vs remote blob
    by `updated` timestamp (newer wins, whole-blob, no entry merge).
    Starts onSnapshot live listener for cross-device updates.
  - Toggle OFF: pauses sync only. Stays signed in to Firebase Auth.
    Firestore data untouched. Local data untouched.
  - "Delete Cloud Data & Sign Out" (separate destructive action, only
    visible when signed in): deletes users/{uid}/vault/data, calls
    signOut() (this device only — does NOT affect other signed-in
    devices), clears cloud-related localStorage keys. Local vault
    entries untouched. Confirm dialog explains other devices may
    auto re-upload.
- Sync triggers: scheduleSync() (800ms debounce) called from
  saveEntries() — fires on every save/edit/delete. checkCloudOnUnlock()
  runs on PIN unlock and biometric unlock — pulls newer remote blob if
  one exists (handles offline/locked period changes). 'online' window
  event also triggers scheduleSync().
- CloudSync.restore() on cold init re-attaches onAuthStateChanged if
  vault_cloud_sync==='1', restoring the live listener without
  re-prompting sign-in (Firebase Auth persists session per-device).
- Live listener (onSnapshot) guards: only acts if state.unlocked &&
  state.currentPIN set — locked vault ignores remote pushes until
  unlocked, at which point checkCloudOnUnlock() catches up anyway.
- Home header shows cloud status icon only when sync ON: 🔄 syncing /
  ☁️ synced / ❌ failed / 📴 offline. Settings shows account email +
  "Synced X ago" / "Sync failed — tap to retry" (tap retries) /
  "Offline — changes saved locally".
- Conflict resolution: last-write-wins by blob-level `updated`
  timestamp, both on initial enable and on every listener push —
  whole-blob replace, no per-entry merging (matches roadmap).
- Cross-device real-time coordination (presence, tombstones, merge UI)
  explicitly out of scope — discussed and deferred as a much larger
  redesign; last-write-wins is acceptable for a 1-user/few-device app.

CRITICAL BUG FOUND & FIXED — Test 2 (Initial Reconcile)
---------------------------------------------------------
SEVERITY: Data loss. FIXED this session, but caused real data loss
during testing (recovered via .vault backup import — see below).

Repro: Device 1 (real entries, synced earlier). Device 2: same PIN
entered for the FIRST time (fresh vault → setup flow runs
saveEntries() with empty entries[], which sets vault_local_updated
to "now"). Enable sync on Device 2.

Bug: reconcile() compared only `remote.updated > local.updated`.
Device 2's vault_local_updated (just now, from PIN setup) was NEWER
than Device 1's cloud updated (set earlier) — even though Device 2's
entries[] was empty. So Device 2 treated itself as "newer" and PUSHED
its empty blob to Firestore, overwriting Device 1's real data. The
live listener then pushed that empty blob down to Device 1 too —
BOTH devices ended up empty.

Fix (CloudSync.reconcile(), startListener(), checkCloudOnUnlock()):
- reconcile() now also checks local/remote emptiness (decrypts remote
  to check entries.length). Empty-local + non-empty-remote ALWAYS
  pulls down regardless of timestamps. Non-empty-local + empty-remote
  ALWAYS pushes up regardless of timestamps. New helper
  CloudSync._isRemoteEmpty(remote).
- startListener() onSnapshot callback and checkCloudOnUnlock(): if a
  remote update decrypts to an empty array AND local state.entries is
  non-empty, the update is IGNORED (console.warn logged) rather than
  applied — prevents an empty remote push from silently wiping a
  populated local vault, even after initial reconcile.

Recovery performed: restored Device 1 from a recent .vault export via
Settings → Import Backup. Import → saveEntries() → fresh
vault_local_updated → pushed real data back to Firestore. Device 2
(empty local, non-empty remote) then correctly pulled the restored
data via the fixed logic. Both devices now have correct data.

LESSON: timestamp-only "newer wins" is unsafe when one side can be
"newer but empty" (e.g. any fresh PIN setup). Emptiness must be
checked alongside timestamps for any blob-replacement decision.
